## 1. Render Tree Architecture - Deep Dive

### Option A: Separate Render Tree (Chrome/Blink Model)

```
DOM Tree (owned by Tempest)         Render Tree (owned by Iris)
┌─────────────────┐                 ┌─────────────────────┐
│ <html>          │                 │ RenderBlock (root)  │
│  ├─ <head>      │   ──style──▶    │  └─ RenderBlock     │
│  └─ <body>      │   ──attach──▶   │      ├─ RenderText  │
│      ├─ <div>   │                 │      └─ RenderInline│
│      └─ <p>     │                 │          └─ ...     │
└─────────────────┘                 └─────────────────────┘
```

**Pros:**
- Clean separation of concerns
- Render tree can skip `display: none` elements entirely
- Easier to optimize (render tree is layout-focused)
- Can have multiple render objects per DOM node (e.g., `::before`, `::after`)
- Incremental updates are more focused

**Cons:**
- Extra memory for parallel tree
- Must keep trees synchronized
- More complex invalidation logic

### Option B: Integrated Tree (Servo Model)

```
┌─────────────────────────────────────┐
│ DOM Node + Layout Data              │
│  ├─ element: Element                │
│  ├─ computed_style: ComputedStyle   │
│  ├─ layout_box: Option<LayoutBox>   │
│  └─ paint_data: Option<PaintInfo>   │
└─────────────────────────────────────┘
```

**Pros:**
- Single source of truth
- Less memory overhead
- Simpler traversal (one tree)
- Natural for incremental updates

**Cons:**
- Harder to parallelize (tree is shared)
- `display: none` nodes still exist in tree
- Mixing concerns (parsing/layout/paint)

### 🎯 **Recommendation: Separate Render Tree**

Given that:
1. Tempest owns the DOM
2. Iris owns rendering
3. We want clean module boundaries

A **separate render tree** makes the most sense. Iris receives "style events" from Tempest and builds its own optimized tree.

## 2. Incremental vs. Parallel - Analysis

### Incremental Rendering

```
Frame 1: Full layout (expensive)
Frame 2: Only dirty subtree re-layouted (cheap)
Frame 3: Only dirty subtree re-layouted (cheap)
```

**Key insight:** Incremental is about **invalidation** - knowing what changed and only recomputing that.

### Parallel Rendering

```
Frame N: 
  Thread 1: Layout left subtree
  Thread 2: Layout right subtree
  Thread 3: Paint background layer
  Thread 4: Paint foreground layer
```

**Key insight:** Parallel is about **independence** - dividing work that doesn't depend on each other.

### 🎯 **Recommendation: Start Incremental, Design for Parallel**

1. **Phase 1 (MVP):** Single-threaded, full layout each frame
2. **Phase 2:** Add dirty tracking, incremental layout
3. **Phase 3:** Parallel paint (layers are independent)
4. **Phase 4:** Parallel layout (with careful dependency tracking)

The key is to **design data structures that support parallel later**:
- Immutable where possible
- Clear ownership boundaries
- No shared mutable state between siblings

### Event-Based Architecture

```

│                      TEMPEST                                  │

│  Lexis parses HTML/CSS → DOM changes → Style events          │
└────────────────────────┬─────────────────────────────────────┘
                         │ StyleEvent stream

┌──────────────────────────────────────────────────────────────┐
│                        IRIS                                   │

│  │   NodeMoved(node_id, new_parent, new_index)             │ │
│  │   StyleChanged(node_id, property, value)                │ │
│  │   TextContentChanged(node_id, text)                     │ │
│  │   AttributeChanged(node_id, attr, value)                │ │
│  └─────────────────────────────────────────────────────────┘ │
└────────────────────────┬─────────────────────────────────────┘
                         │ RenderCommand / SubmitBuffer
                         ▼
┌──────────────────────────────────────────────────────────────┐
│                        PRISM                                  │
│                                                              │
│  Composites layers → Displays to screen                      │
│  Sends WindowEvent back to Iris                             │
└──────────────────────────────────────────────────────────────┘
```

### Who Owns What?

| Component | Owns | Receives | Sends |
|-----------|------|----------|-------|
| **Tempest** | DOM tree, navigation state | Window events (from Iris) | Style events (to Iris) |
| **Iris** | Render tree, layer tree, frame scheduling | Style events, Window events | Render commands |
| **Prism** | Windows, surfaces, compositor | Render commands | Window events |

### 🎯 **Key Design Decision: Iris Owns the Window**

Since Iris controls:
- What gets painted
- When frames are submitted
- Layer compositing

**Iris should own the Prism window.** Tempest communicates with Iris, not directly with Prism.

```
User Input → Prism → Iris → Tempest (for DOM/JS handling)
                  ↓
            Iris → Prism (render result)
```

### Sync vs Async Complexity

**Synchronous:**
```ritz
# Simple - Tempest controls everything
fn handle_dom_change()
    dom.mutate()
    iris.apply_style_change(node_id, style)  # Marks dirty
    iris.layout()                             # Blocks, does full layout
    iris.paint()                              # Blocks, sends to Prism
    # Done, frame visible
```
- **Complexity:** Low. Linear flow.
- **Problem:** If JS does 1000 DOM mutations, you layout 1000 times (unless Tempest batches).

**Async with callbacks:**
```ritz
# More state management
fn handle_dom_change()
    dom.mutate()
    iris.queue_style_change(node_id, style)
    # Returns immediately, layout happens later

fn on_layout_complete(result: LayoutResult)
    # Called when Iris finishes
    # But what if more mutations came in? Need to track generations.
```
- **Complexity:** Medium. Need state machine, generation tracking.
- **Benefit:** Never blocks Tempest's event loop.

**Frame-based (what Chrome/Firefox do):**
```ritz
# Tempest event loop
loop
    process_events()           # Mouse, keyboard, network
    execute_js_tasks()         # Run pending JS
    
    # At 60fps boundary (or requestAnimationFrame):
    iris.flush_pending()       # Apply all queued mutations
    iris.layout_if_dirty()     # Layout only if needed
    iris.paint()               # Generate frame
    wait_for_vsync()
```
- **Complexity:** Low-Medium. Just need to batch mutations.

1. **Natural batching:** 100 JS DOM mutations → 1 layout
2. **Vsync aligned:** Matches display refresh rate
3. **Predictable timing:** `requestAnimationFrame` works naturally
4. **No wasted work:** Only layout/paint when something changed
