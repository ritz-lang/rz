### Key Decisions That Affect Lexis

1. **DOM Ownership** → **Tempest owns the DOM**, not Lexis or Iris
   - Lexis parses HTML → produces DOM nodes → Tempest stores them
   - Tempest is the orchestrator

2. **Style Computation** → **Lexis handles cascade & computed styles**
   - Lexis: Parse CSS, selector matching, cascade, specificity, inheritance → **ComputedStyle**
   - Iris: Takes ComputedStyle → Layout → Paint
   - Clean split at the ComputedStyle boundary

3. **Iris is an embedded library** in Tempest, not a separate process
   - IPC happens between Iris↔Prism, not Tempest↔Iris

4. **Frame-based batching** for DOM mutations
   - Tempest queues style changes, Iris processes them once per frame

5. **Prism provides primitives** that everyone uses
   - `Color`, `Rect`, `Point`, `Size` from `prism.api`

### What Lexis Must Provide

**To Tempest:**
1. **HTML Parser** → DOM nodes
2. **CSS Parser** → Parsed stylesheets
3. **Selector Matching** → Which rules apply to which elements
4. **Cascade** → Resolve specificity, inheritance
5. **ComputedStyle** → Fully resolved values per node

**StyleEvent Stream** (what Tempest sends to Iris):
```ritz
enum StyleEvent
    NodeCreated { id: NodeId, parent: NodeId, index: u32, tag: Tag }
    NodeRemoved { id: NodeId }
    NodeMoved { id: NodeId, new_parent: NodeId, new_index: u32 }
    StyleChanged { id: NodeId, style: ComputedStyle }
    TextChanged { id: NodeId, text: StrView }
```

## Streaming + Event-Based → DOM

You're right — this is exactly how real browsers work. Here's my thinking:

```
HTML Source (streaming bytes)
         │
         ▼
┌─────────────────────┐
│   HTML TOKENIZER    │  ← State machine, emits tokens as they complete
│   (streaming)       │
└──────────┬──────────┘
           │ Token stream (StartTag, EndTag, Character, Comment, etc.)
           ▼
┌─────────────────────┐
│   TREE BUILDER      │  ← Implements HTML5 insertion modes
│   (event emitter)   │
└──────────┬──────────┘
           │ DOM Events (NodeCreated, TextInserted, etc.)
           ▼
┌─────────────────────┐
│   TEMPEST           │  ← Receives events, owns DOM
│   (DOM owner)       │
└─────────────────────┘
```

### Why Event-Based?

1. **Incremental rendering** — Tempest can start sending StyleEvents to Iris *while* parsing continues
2. **Memory efficient** — Don't need whole document in memory before acting
3. **Natural for network** — Bytes arrive incrementally from Valet
4. **JS interop** — `document.write()` can inject into the token stream (future)

### Lexis Event Protocol

```ritz
# Events Lexis emits to Tempest during parsing
enum DOMEvent
    # Document lifecycle
    DocumentStart
    DocumentEnd
    
    # Structure
    ElementStart { tag: StrView, attrs: Vec<Attribute> }
    ElementEnd { tag: StrView }
    Text { content: StrView }
    Comment { content: StrView }
    
    # For <style> blocks and <link rel="stylesheet">
    StylesheetParsed { rules: Vec<CSSRule> }
    
    # Errors (to debug console)
    ParseError { message: StrView, line: u32, col: u32 }
```

### CSS Selector Engine — "Ritzy Mode" 🎩

Oh we're going *full performance* on this. Here's the plan:

```
┌─────────────────────────────────────────────────────────────┐
│                   SELECTOR ENGINE                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. RULE INDEXING (at stylesheet parse time)                │
│     ┌─────────────────────────────────────────────────────┐ │
│     │ - ID index:    HashMap<StrView, Vec<RuleIndex>>     │ │
│     │ - Class index: HashMap<StrView, Vec<RuleIndex>>     │ │
│     │ - Tag index:   HashMap<StrView, Vec<RuleIndex>>     │ │
│     │ - Universal:   Vec<RuleIndex>                       │ │
│     └─────────────────────────────────────────────────────┘ │
│                                                             │
│  2. BLOOM FILTER (fast rejection)                           │
│     ┌─────────────────────────────────────────────────────┐ │
│     │ - Each element has ancestor filter                  │ │
│     │ - Quickly reject "div.foo .bar" if no .foo ancestor │ │
│     │ - Cascading filter construction during tree build   │ │
│     └─────────────────────────────────────────────────────┘ │
│                                                             │
│  3. RIGHT-TO-LEFT MATCHING                                  │
│     ┌─────────────────────────────────────────────────────┐ │
│     │ - Match rightmost (key) selector first              │ │
│     │ - Walk ancestors to verify combinators              │ │
│     │ - Most selectors fail on key, so this is fastest    │ │
│     └─────────────────────────────────────────────────────┘ │
│                                                             │
│  4. SPECIFICITY + CASCADE                                   │
│     ┌─────────────────────────────────────────────────────┐ │
│     │ - Sort matching rules by (origin, specificity, order)│ │
│     │ - Apply in cascade order                            │ │
│     │ - Handle !important                                  │ │
│     └─────────────────────────────────────────────────────┘ │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

This is the same architecture Servo/Stylo uses. It's **fast as fuck**.
