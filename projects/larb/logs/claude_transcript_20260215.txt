[2026-02-15T21:47:01.346134] Claude: You're having the exact realization that many projects eventually hit. Let me break down the tradeoffs:

## The Core Problem

You have:
1. **Multiple repos** that depend on each other (`ritz`, `squeeze`, `cryptosec`, etc.)
2. **Nested submodules** creating diamond dependency hell (`ritz` appearing 5 times in different places)
3. **Multiple agent sessions** each treating their workspace as the root
4. **Cross-repo changes** that need to be atomic

## Option A: Monorepo (What You're Screaming About)

```
ritz-lang/
├── .git/                    # ONE repo
├── ritz/                    # compiler + ritzlib
├── ritzunit/
├── squeeze/
├── cryptosec/
├── valet/
├── zeus/
├── mausoleum/
├── tome/
├── spire/
├── harland/
├── larb/
└── angelo/
```

**Pros:**
- Atomic cross-project commits
- No submodule sync issues ever
- Single source of truth
- `RITZ_PATH` just points here
- Agent sessions can scope to subdirs but share code
- Refactoring across boundaries is trivial

**Cons:**
- Larger clone (but sparse checkout helps)
- Single version of everything (but that's actually what you want)
- GitHub doesn't show them as "separate projects" (cosmetic)

## Option B: Parent Repo with Submodules (`ritz-dev`)

```
ritz-dev/
├── .git/
├── .gitmodules              # points to all the repos
├── ritz/       → github.com/ritz-lang/ritz
├── squeeze/    → github.com/ritz-lang/squeeze
└── ...
```

**Pros:**
- Keeps separate repos for "open source optics"
- Can pin specific versions
- Selective initialization

**Cons:**
- Submodule update dance for every cross-repo change
- Still have sync headaches, just moved up a level
- Agents need to understand they're in a submodule

## My Take

**Go monorepo.** Here's why:

1. **You're making the language** — all these projects evolve together in lockstep. There's no "I want squeeze 1.2 with ritz 1.5" — they move as a unit.

2. **No Concessions Doctrine applies here too** — submodules are a concession to "projects should be separate." But should they? They're all ritz-lang.

3. **Agent sessions** become scoped views into ONE repo. An agent working on `squeeze` just focuses on that dir but can freely touch `ritz/ritzlib` when needed.

4. **Cross-repo PRs become normal PRs** — "Add gzip support to squeeze and use it in valet" is one commit, one review, one merge.

5. **The "separate repos" thing is cosmetic** — who are you maintaining that for? External contributors? They can still work in subdirs.

## Migration Path

```bash
# Create the monorepo
mkdir ritz-mono && cd ritz-mono

# Pull in each project's history (optional: preserve it)

    git subtree add --prefix=$repo git@github.com:ritz-lang/$repo.git main
done

```
