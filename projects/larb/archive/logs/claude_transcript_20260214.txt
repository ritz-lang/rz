## Introduction

MVRSPT (Model-View-Repository-Service-Presenter-Tests) is a six-layer architectural pattern that provides clean separation of concerns, particularly suited for applications with external dependencies and complex business logic. It extends traditional MVC/MVP patterns by adding explicit Repository, Service, and Test layers.

### When to Use MVRSPT
- Applications with external API dependencies
- Systems requiring database abstraction
- Projects where testability is crucial
- Applications with complex business logic
- AI-driven applications with tool orchestration
- Systems that may need to swap data sources

## The Six Layers

### 🗄️ Model (M) - Data Structure
**Purpose:** Pure data representation without business logic

**Responsibilities:**
- Define data schemas and structures
- Represent domain objects
- Handle data validation at the database level
- Maintain relationships between entities

**Examples:**
```python
# Django Model
class Task(models.Model):
    title = models.CharField(max_length=200)
    description = models.TextField(blank=True)
    completed = models.BooleanField(default=False)
    user = models.ForeignKey(User, on_delete=models.CASCADE)
    created_at = models.DateTimeField(auto_now_add=True)
```

### 👁️ View (V) - User Interface
**Purpose:** Handle presentation and user interaction

**Responsibilities:**
- Render user interfaces
- Handle user input
- Format data for display
- Manage UI state (not business state)

**Examples:**
```jsx
// React Component
function TaskList({ tasks, onTaskComplete }) {
    return (
        <ul>
            {tasks.map(task => (
                <TaskItem 
                    key={task.id} 
                    task={task} 
                    onComplete={onTaskComplete}
                />
            ))}
        </ul>
    );
}
```

### 🏭 Repository (R) - Data Access Abstraction
**Purpose:** Abstract HOW data is accessed and stored

**Responsibilities:**
- Provide consistent interface for data operations
- Implement caching strategies
- Handle database transactions
- Manage connection pooling
- Add audit logging
- Implement rate limiting for external APIs

**Key Principle:** The Repository is the ONLY layer that knows about the actual data source.

**Examples:**
```python
class TaskRepository:
    """Repository pattern for Task data access"""
    
    def create_task(self, user, title, description=None):
        """Factory function for task creation"""
        return Task.objects.create(
            user=user,
            title=title,
            description=description
        )
    
    def get_user_tasks(self, user, completed=None):
        """Retrieve tasks with optional filtering"""
        queryset = Task.objects.filter(user=user)
        if completed is not None:
            queryset = queryset.filter(completed=completed)
        return queryset.order_by('-created_at')
    
    def update_task(self, task_id, **updates):
        """Update task attributes"""
        Task.objects.filter(id=task_id).update(**updates)
        return self.get_task_by_id(task_id)
    
    def bulk_create_tasks(self, tasks_data):
        """Efficient bulk creation"""
        tasks = [Task(**data) for data in tasks_data]
        return Task.objects.bulk_create(tasks)

# Advanced Repository with Caching
class CachedTaskRepository(TaskRepository):
    def __init__(self, cache_backend):
        self.cache = cache_backend
    
    def get_user_tasks(self, user, completed=None):
        cache_key = f"user_tasks:{user.id}:{completed}"
        cached = self.cache.get(cache_key)
        if cached:
            return cached
        
        tasks = super().get_user_tasks(user, completed)
        self.cache.set(cache_key, tasks, timeout=300)
        return tasks
```

### ⚙️ Service (S) - Business Logic
**Purpose:** Orchestrate business rules and workflows

**Responsibilities:**
- Implement business rules
- Coordinate between multiple repositories
- Handle complex workflows
- Enforce business constraints
- Manage transactions across repositories

**Examples:**
```python
class TaskService:
    """Service layer for task business logic"""
    
    def __init__(self, task_repo, notification_repo=None):
        self.task_repo = task_repo
        self.notification_repo = notification_repo
    
    def create_task_for_user(self, user, title, description=None):
        """Create task with business validation"""
        # Business rule: Title must be meaningful
        if not title or len(title.strip()) < 3:
            raise ValidationError("Task title must be at least 3 characters")
        
        # Business rule: User cannot have more than 100 pending tasks
        pending_count = self.task_repo.count_pending_tasks(user)
        if pending_count >= 100:
            raise BusinessRuleViolation("Maximum pending tasks limit reached")
        
        # Create the task
        task = self.task_repo.create_task(
            user=user,
            title=title.strip(),
            description=description
        )
        
        # Send notification if configured
        if self.notification_repo:
            self.notification_repo.notify_task_created(user, task)
        
        return task
    
    def complete_task(self, user, task_id):
        """Complete a task with business logic"""
        task = self.task_repo.get_task_by_id(task_id)
        
        # Business rule: User can only complete their own tasks
        if task.user_id != user.id:
            raise PermissionDenied("Cannot complete another user's task")
        
        # Business rule: Cannot complete already completed task
        if task.completed:
            raise BusinessRuleViolation("Task already completed")
        
        return self.task_repo.update_task(task_id, completed=True)
    
    def get_productivity_metrics(self, user, date_range):
        """Complex business logic spanning multiple repositories"""
        tasks = self.task_repo.get_tasks_in_range(user, date_range)
        completed = [t for t in tasks if t.completed]
        
        return {
            'total_tasks': len(tasks),
            'completed_tasks': len(completed),
            'completion_rate': len(completed) / len(tasks) if tasks else 0,
            'average_completion_time': self._calculate_avg_time(completed)
        }
```

### 🎭 Presenter (P) - Intelligent Controller
**Purpose:** Coordinate user intent with business services

**Responsibilities:**
- Receive and interpret user requests
- Call appropriate services
- Format responses for views
- Handle errors gracefully
- Manage session state
- In AI apps: LLM decision-making

**Examples:**
```python

        self.service = TaskService(
            task_repo=TaskRepository(),
            notification_repo=NotificationRepository()

            task = self.service.create_task_for_user(
                user=request.user,
                title=request.data.get('title'),
                description=request.data.get('description')
            )
            serializer = TaskSerializer(task)
            return Response(serializer.data, status=201)
        except ValidationError as e:
            return Response({'error': str(e)}, status=400)
        except BusinessRuleViolation as e:
            return Response({'error': str(e)}, status=409)

# AI Application Presenter (LLM-based)
class AITaskPresenter:
    """LLM acts as presenter, interpreting natural language"""
    
    def __init__(self, llm, task_service):
        self.llm = llm
        self.service = task_service
    
    async def handle_user_message(self, user, message):
        """Interpret user intent and execute appropriate action"""
        intent = await self.llm.analyze_intent(message)
        
        if intent.action == "create_task":
            task = self.service.create_task_for_user(
                user=user,
                title=intent.params['title'],
                description=intent.params.get('description')
            )
            return f"Created task: {task.title}"
        
        elif intent.action == "list_tasks":
            tasks = self.service.get_user_tasks(user)
            return self._format_task_list(tasks)
        
        # LLM as intelligent router
        return await self.llm.generate_response(message)

## Architecture Flow

### Request Flow
```
User Action → View → Presenter → Service → Repository → Model → Database
     ↑                    ↓           ↓          ↓         ↓        ↓
     └────────────── Response ←──────┴──────────┴─────────┴────────┘
```

### Layer Interactions
```
┌─────────────────────────────────────────────┐
│                   View                       │
│  (React, HTML, Mobile App, CLI, Slack Bot)  │
└─────────────────────┬───────────────────────┘
                      │ HTTP/GraphQL/WebSocket
┌─────────────────────▼───────────────────────┐
│                 Presenter                    │
│    (Controller, ViewSet, GraphQL Resolver)  │
│         (Or LLM in AI applications)         │
└─────────────────────┬───────────────────────┘
                      │ Method Calls
┌─────────────────────▼───────────────────────┐
│                  Service                     │
│       (Business Logic, Workflows)           │
└─────────────────────┬───────────────────────┘
                      │ Interface Calls
┌─────────────────────▼───────────────────────┐
│                Repository                    │
│   (Data Access, Caching, External APIs)     │
└─────────────────────┬───────────────────────┘
                      │ ORM/SQL/API
┌─────────────────────▼───────────────────────┐
│                   Model                      │
│        (Domain Objects, Entities)           │
└─────────────────────┬───────────────────────┘
                      │
┌─────────────────────▼───────────────────────┐
│              Data Source                     │
│   (PostgreSQL, MongoDB, Redis, External API) │
└──────────────────────────────────────────────┘
```

## Benefits

### 🔄 Flexibility
- **Backend Independence**: Switch from PostgreSQL to MongoDB without changing services
- **API Agility**: Replace external APIs without modifying business logic
- **Caching Freedom**: Add Redis caching without touching controllers
- **Testing Options**: Use in-memory database for tests, real database for production

### 🧪 Testability
- **Isolated Testing**: Test business logic without database
- **Fast Tests**: Mock repositories for instant unit tests
- **Reliable Tests**: No test pollution from shared database state
- **Coverage**: Each layer can achieve 100% test coverage independently

### 📦 Maintainability
- **Clear Boundaries**: Obvious where each piece of code belongs
- **Single Responsibility**: Each layer has one clear job
- **Reduced Coupling**: Changes don't cascade through the system
- **Parallel Development**: Teams can work on different layers simultaneously

### 🚀 Scalability
- **Performance Optimization**: Add caching at repository layer
- **Bulk Operations**: Implement efficient batch processing
- **Connection Pooling**: Manage resources at repository level
- **Microservices Ready**: Each service can become its own microservice

## Comparison with Traditional Patterns

### MVC (Model-View-Controller)
```
MVC:
Controller → Model ← View
(Mixed concerns, hard to test)

MVRSPT:
View → Presenter → Service → Repository → Model
(Clear separation, each layer testable)
```

### Problems with Traditional MVC:
- Controller often contains business logic
- Direct database access from controllers
- Hard to test without full stack
- Switching databases requires controller changes
- No clear place for caching logic

### MVRSPT Solutions:
- Business logic isolated in Service layer
- Database access abstracted by Repository
- Each layer testable with mocks
- Database changes only affect Repository
- Clear caching layer in Repository

## Best Practices

### 1. Repository Design
- Keep repositories focused on data access only
- Use interfaces/protocols for repositories
- Implement caching transparently
- Handle pagination at repository level
- Don't leak ORM specifics

### 2. Service Design
- Services should be framework-agnostic
- Coordinate multiple repositories when needed
- Implement transaction management
- Throw business-specific exceptions
- Keep services stateless

### 3. Presenter Design
- Keep presenters thin
- Handle HTTP concerns only
- Delegate all logic to services
- Format responses appropriately
- Handle errors gracefully

### 4. Testing Strategy
- Mock repositories for service tests
- Mock services for presenter tests
- Use real implementations for integration tests
- Test each layer's public interface
- Maintain high coverage per layer

### 5. Dependency Injection
```python
# Good: Inject dependencies
class TaskService:
    def __init__(self, task_repo, user_repo):
        self.task_repo = task_repo
        self.user_repo = user_repo

# Bad: Hard-coded dependencies
class TaskService:
    def __init__(self):
        self.task_repo = TaskRepository()  # Hard to test
        self.user_repo = UserRepository()  # Can't mock
```

### 6. Error Handling
```python
# Define business-specific exceptions
class BusinessRuleViolation(Exception):
    pass

class ResourceNotFound(Exception):
    pass

class PermissionDenied(Exception):
    pass

# Service throws business exceptions
def complete_task(self, user, task_id):
    task = self.repo.find(task_id)
    if not task:
        raise ResourceNotFound(f"Task {task_id} not found")
    if task.user_id != user.id:
        raise PermissionDenied("Cannot modify another user's task")
    # ...

# Presenter catches and formats
def update(self, request, task_id):
    try:
        task = self.service.complete_task(request.user, task_id)
        return Response(TaskSerializer(task).data)
    except ResourceNotFound:
        return Response({'error': 'Not found'}, status=404)
    except PermissionDenied:
        return Response({'error': 'Forbidden'}, status=403)
```

## FAQ

### Q: Isn't this overengineering for simple apps?
**A:** For truly simple CRUD apps, MVRSPT might be overkill. However, most apps grow in complexity over time. Starting with clean separation makes growth painless. You can also adopt MVRSPT gradually, starting with just Repository pattern for testing benefits.

### Q: How does this relate to Clean Architecture or Hexagonal Architecture?
**A:** MVRSPT shares principles with both:
- Like Clean Architecture, it separates concerns into layers
- Like Hexagonal Architecture, it uses ports (interfaces) and adapters (repositories)
- MVRSPT is more prescriptive about layer names and responsibilities

### Q: Can I skip the Repository layer if I'm using an ORM?
**A:** The Repository still adds value even with an ORM:
- Makes testing much easier
- Provides a place for caching logic
- Allows you to optimize queries without changing services
- Makes it possible to switch ORMs or databases

### Q: Where does authentication/authorization fit?
**A:** Authentication typically happens at the Presenter layer (checking who the user is), while authorization can be split:
- Simple permission checks in Presenter
- Business rule authorization in Service layer
- Data-level authorization in Repository

### Q: How do I handle transactions across multiple repositories?
**A:** The Service layer should coordinate transactions:
```python
class OrderService:
    def create_order(self, user, items):
        with self.db.transaction():
            order = self.order_repo.create(user)
            for item in items:
                self.order_item_repo.create(order, item)
                self.inventory_repo.decrease(item.product_id, item.quantity)
            return order
```

### Q: Should every model have its own repository?
**A:** Not necessarily. Group related models:
- UserRepository might handle User and Profile models
- OrderRepository might handle Order and OrderItem models
- Keep repositories cohesive around business concepts

### Q: How does this work with GraphQL?
**A:** GraphQL resolvers act as the Presenter layer:
```javascript
const resolvers = {
    Query: {
        tasks: (parent, args, context) => {
            return taskService.getUserTasks(context.user);
        }
    },
    Mutation: {
        createTask: (parent, { title }, context) => {
            return taskService.createTask(context.user, title);
        }
    }
};
```

### Q: What about event-driven architectures?
**A:** Events can be published from the Service layer:
```python
class TaskService:
    def complete_task(self, user, task_id):
        task = self.repo.complete(task_id)
        self.event_bus.publish('task.completed', {
            'task_id': task_id,
            'user_id': user.id
        })
        return task
```

## Conclusion

MVRSPT provides a robust, testable, and maintainable architecture for modern applications. By clearly separating concerns into six distinct layers, it enables:

- Independent testing of each layer
- Easy swapping of implementations
- Clear boundaries for team responsibilities
- Predictable location for new features
- Sustainable growth as applications scale

The pattern is particularly valuable for:
- Applications with complex business logic
- Systems requiring high test coverage
- Projects with external dependencies
- Teams practicing TDD/BDD
- Applications that may need to pivot technologies

Start small by introducing the Repository pattern for better testing, then gradually adopt other layers as your application grows.

### The Six Layers in Ritz

| Layer | Ritz Module | Responsibility |
|-------|-------------|----------------|
| **M** - Model | `models/` | Pure data structs, no logic |
| **V** - View | `views/` | Templates, HTML rendering |
| **R** - Repository | `repos/` | Data access abstraction (Mausoleum/Tome interface) |
| **S** - Service | `services/` | Business logic, workflows |
| **P** - Presenter | `presenters/` | Route handlers, request/response |
| **T** - Tests | `tests/` | Unit + integration, mocks at every layer |

struct WikiPage
    id: Uuid
    slug: StrView
    title: String
    content: String
    parent_id: Option<Uuid>
    version: u64
    created_at: Timestamp
    updated_at: Timestamp

// repos/wiki_repo.ritz — Data access abstraction
struct WikiRepository
    db: @Mausoleum
    cache: @Tome

impl WikiRepository
    fn new(db: @Mausoleum, cache: @Tome) -> WikiRepository
        WikiRepository { db, cache }
    
    fn find_by_slug(self: @WikiRepository, slug: StrView) -> Option<WikiPage>
        // Check cache first
        let cache_key = "page:" + slug
        if let Some(page) = self.cache.get(cache_key)
            return Some(page)
        
        // Fall back to database
        let page = self.db.query()
            .kind("wiki_page")
            .where("slug", slug)
            .first()?
        
        // Populate cache
        self.cache.set(cache_key, page, ttl: 300)
        Some(page)
    
    fn create(self:& WikiRepository, page: WikiPage) -> Result<WikiPage, Error>
        self.db.insert(page)?
        self.cache.invalidate("page:" + page.slug)
        Ok(page)

// services/wiki_service.ritz — Business logic
struct WikiService
    repo: WikiRepository
    search: SearchService

impl WikiService
    fn create_page(self:& WikiService, author: User, title: StrView, content: StrView) -> Result<WikiPage, Error>
        // Business rule: title must be meaningful
        if title.len() < 3
            return Err(ValidationError("Title must be at least 3 characters"))
        
        // Business rule: generate slug from title
        let slug = slugify(title)
        
        // Business rule: slug must be unique
        if self.repo.find_by_slug(slug).is_some()
            return Err(ConflictError("Page with this title already exists"))
        
        let page = WikiPage {
            id: Uuid.new_v4(),
            slug: slug,
            title: String.from(title),
            content: String.from(content),
            parent_id: None,
            version: 1,
            created_at: Timestamp.now(),
            updated_at: Timestamp.now(),
        }
        
        // Create in repository
        self.repo.create(page)?
        
        // Index for search
        self.search.index_page(page)
        
        Ok(page)
    
    fn get_page_tree(self: @WikiService, root_slug: StrView) -> Result<Vec<WikiPage>, Error>
        // Complex business logic: build hierarchical tree
        let root = self.repo.find_by_slug(root_slug)?
        self.repo.get_descendants(root.id)

### Key Insight: Repository as the Seam

The Repository layer is where Spire gets its power:

```ritz
// The trait that ALL repositories implement
trait Repository<T>
    fn find(self: @Self, id: Uuid) -> Option<T>
    fn find_all(self: @Self, query: Query) -> Vec<T>
    fn create(self:& Self, item: T) -> Result<T, Error>
    fn update(self:& Self, item: T) -> Result<T, Error>
    fn delete(self:& Self, id: Uuid) -> Result<(), Error>

// Mausoleum implementation (persistent)
struct MausoleuRepository<T>
    db: @Mausoleum

impl<T> Repository<T> for MausoleuRepository<T>
    fn find(self: @Self, id: Uuid) -> Option<T>
        self.db.get(id)
    // ...

// Tome implementation (in-memory/cache)
struct TomeRepository<T>
    cache: @Tome

impl<T> Repository<T> for TomeRepository<T>
    fn find(self: @Self, id: Uuid) -> Option<T>
        self.cache.get(id)
    // ...

// Mock implementation (testing)
struct MockRepository<T>
    items: HashMap<Uuid, T>
    should_fail: bool

impl<T> Repository<T> for MockRepository<T>
    fn find(self: @Self, id: Uuid) -> Option<T>
        if self.should_fail
            return None
        self.items.get(id)
    // ...
```

## `ritz-lint` Design Proposal

### Core Concept

An AI-powered linter that understands **idiomatic Ritz patterns** and flags C-style code. It would work in two modes:

1. **Rule-based** (fast, deterministic) - Catches obvious issues
2. **AI-assisted** (deeper, contextual) - Catches semantic issues

### Rule-Based Checks (No AI needed)

| Rule | Detects | Suggests |
|------|---------|----------|
| `no-c-strings` | `c"..."` literals | Use `"..."` (StrView) |
| `no-old-attributes` | `@test`, `@inline` | Use `[[test]]`, `[[inline]]` |
| `no-keyword-ops` | `and`, `or`, `not` | Use `&&`, `\|\|`, `!` |
| `no-raw-pointers-in-signatures` | `fn foo(x: *T)` | Use `x: T`, `x:& T`, or `x:= T` |
| `no-mut-keyword` | `*mut T` in params | Use `:&` modifier |
| `prefer-option` | `if ptr != null` | Use `Option<T>` + match |

### AI-Assisted Checks (Contextual)

| Check | Example | Why AI Needed |
|-------|---------|---------------|
| **Ownership inference** | "This function takes `*Data` but never mutates it" | Requires data flow analysis |
| **Borrow vs move** | "Should this be `:&` or `:=`?" | Depends on usage pattern |
| **Null pattern detection** | "This null check could be `Option<T>`" | Semantic understanding |
| **Resource cleanup** | "Missing `defer` for this file handle" | Context-dependent |

### Implementation Options

**Option A: Pure Ritz (ideal but harder)**
```
ritz-lint/
├── src/
│   ├── main.ritz       # CLI entry
│   ├── parser.ritz     # Reuse ritz1 parser
│   ├── rules/          # Rule implementations
│   └── reporter.ritz   # Output formatting
```

**Option B: Python + AI (faster to build)**
```
ritz-lint/
├── ritz_lint/
│   ├── __init__.py
│   ├── cli.py          # CLI entry
│   ├── parser.py       # Reuse ritz0 parser
│   ├── rules.py        # Deterministic rules
│   └── ai_review.py    # Claude API for deep analysis
```

**Option C: Hybrid (pragmatic)**
- Rule-based checks in Ritz (fast, native)
- AI review via Claude API for PR comments
- GitHub Action wrapper
