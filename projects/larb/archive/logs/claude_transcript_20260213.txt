## Tree-sitter Grammar from Ritz Grammar

**Short answer: Yes, but not automatically.**

The `ritz.grammar` format is a custom PEG-like grammar with semantic actions for AST construction. Tree-sitter uses a different DSL (`grammar.js` in JavaScript/JSON). The conversion requires manual translation, but the structure maps well:

| Ritz Grammar | Tree-sitter Equivalent |
|--------------|------------------------|
| `%tokens` section | `extras`, `keywords`, `token()` |
| `rule : alt1 \| alt2 ;` | `rule_name: $ => choice(...)` |
| `rule*` | `repeat($.rule)` |
| `rule?` | `optional($.rule)` |
| Precedence (implicit) | `prec.left()`, `prec.right()` |
| Semantic actions `{ ... }` | Dropped (tree-sitter is parse-only) |

The indentation-sensitive `INDENT`/`DEDENT` tokens are the tricky part—tree-sitter has limited support for context-sensitive lexing, requiring an external scanner in C.

**1. `:=` (Move) on Copy types = "I'm done with this"**

- Readability (reviewer knows counter isn't used later)
- Optimization (compiler can reuse the register/stack slot immediately)
- Catching bugs (lint: "you marked `counter` as moved but used it again")

## How Do Allocators Work with Borrowing?

This is where it gets interesting. Let me check how this currently works and think through the design:

```ritz
# Current problem: How does an arena return borrowed references?
struct Arena
    base: *u8
    # ...

impl Arena
    # Option A: Return raw pointer (unsafe, caller must track lifetime)
    fn alloc(self:& Arena, bytes: u64) -> *u8

    # Option B: Return reference tied to arena lifetime (needs lifetimes!)
    fn alloc(self:& Arena, bytes: u64) -> @u8   # How long does this live?
```

### The Lifetime Problem

In Rust, you'd write:
```rust
fn alloc<'a>(&'a mut self, bytes: usize) -> &'a mut [u8]
```

The `'a` says "the returned reference lives as long as the borrow of self."

**Ritz wants to avoid explicit lifetimes.** So how do we make this safe?

### Possible Ritz Solutions

**1. Arenas return raw pointers (unsafe, but explicit)**
```ritz
fn alloc(self:& Arena, bytes: u64) -> *u8
    # Caller is responsible for not using after arena.reset() or arena.destroy()
```
This is what C does. Simple, but unsafe.

**2. Arenas are consumed, not borrowed**
```ritz
# Arena hands out "handles" that encode the lifetime relationship
struct ArenaHandle<T>
    arena_id: u64
    offset: u64

impl Arena
    fn alloc<T>(self:& Arena) -> ArenaHandle<T>
        # Returns a handle, not a reference

    fn get<T>(self: Arena, handle: ArenaHandle<T>) -> T
        # Only works if arena is still valid
```
This is like generational indices in ECS systems.

**3. Scoped arenas (structured concurrency for memory)**
```ritz
# Arena scope — everything allocated inside is freed at end of block
with_arena(1024) |arena|
    let buf = arena.alloc(256)    # Returns @u8, valid until block ends
    process(buf)
# buf is invalid here — arena destroyed
```
This is like Zig's arena pattern. The lifetime is tied to the lexical scope.

**4. Ritz infers simple lifetimes**
```ritz
# For simple cases, Ritz could infer that the returned reference
# lives as long as the input reference
fn alloc(self:& Arena, bytes: u64) -> @u8
    # Compiler infers: returned @u8 lives as long as self:& Arena
```

## My Take

For **low-level code** (allocators, kernel dev), use `*T` pointers. This is unsafe territory and should look unsafe.

For **high-level patterns** (arena-based temp allocations), use **scoped arenas** with inferred lifetimes:

```ritz
# High-level (safe, scoped)
with_arena(4096) |arena|
    var users = arena.alloc_slice::<User>(100)
    load_users(users)
```

## Summary: `@` Usage

| Expression | Type | Meaning |
|------------|------|---------|
| `@x` | `@T` | Take a reference to x ("where is x at?") |
| `@mut x` | `@mut T` | Take a mutable reference to x |
| `fn foo(x: T)` | const borrow | Read-only access (signature) |
| `fn foo(x:& T)` | mutable borrow | Can modify (signature) |
| `fn foo(x:= T)` | move | Takes ownership (signature) |
| `-> @T` | return reference | Function returns a reference |

**Key insight:** 
- **In signatures:** `:`, `:&`, `:=` express how params are passed
- **In expressions:** `@x` takes the address/reference of x
- **In types:** `@T` is "reference to T", `*T` is "pointer to T"

## HTTP Protocol Evolution

### HTTP/1.0 (1996)
**The OG - Simple but naive**

- **Connection model**: One request per TCP connection (connect → request → response → close)
- **No pipelining**: Each request waits for full response
- **Headers**: Plain text, no compression
- **Key limitation**: TCP connection overhead is brutal for multi-resource pages

```
Client                    Server
  |----TCP CONNECT-------->|
  |----GET /index.html---->|
  |<---200 OK + body-------|
  |----TCP CLOSE---------->|
  (repeat for every resource)
```

### HTTP/1.1 (1997, refined through 2014)
**The workhorse - Still powers most of the web**

- **Persistent connections**: Keep-alive by default (reuse TCP connection)
- **Pipelining**: Send multiple requests without waiting (but responses must be in order)
- **Chunked transfer**: Stream responses without knowing final size
- **Host header**: Virtual hosting (multiple domains per IP)
- **Caching**: `ETag`, `If-Modified-Since`, `Cache-Control`

**Head-of-line blocking problem**: If request 1 is slow, requests 2-10 wait even if their responses are ready.

```
Client                    Server
  |----TCP CONNECT-------->|
  |----GET /a.html-------->|
  |----GET /b.css--------->|  (pipelined)
  |----GET /c.js---------->|
  |<---200 /a.html---------|  (must come first!)
  |<---200 /b.css----------|
  |<---200 /c.js-----------|
```

**Workarounds**: Domain sharding (open 6 connections to cdn1.example.com, cdn2.example.com...), sprite sheets, bundling.

### HTTP/2 (2015)
**Binary, multiplexed, modern**

Built on Google's SPDY. Single TCP connection, full multiplexing.

**Key concepts**:

| Concept | Description |
|---------|-------------|
| **Streams** | Independent bidirectional sequences of frames within one connection |
| **Frames** | Smallest unit of communication (HEADERS, DATA, PRIORITY, etc.) |
| **Multiplexing** | Multiple streams interleaved on same connection |
| **HPACK** | Header compression (maintains state tables both sides) |
| **Server Push** | Server can send resources client hasn't asked for yet |
| **Priority** | Streams have dependencies and weights |

```
Single TCP Connection
┌─────────────────────────────────────────────┐
│  Stream 1: GET /index.html                  │
│  Stream 3: GET /style.css                   │
│  Stream 5: GET /app.js                      │
│  Stream 7: Server Push /logo.png            │
│                                             │
│  Frames interleaved:                        │
│  [H:1][H:3][D:1][D:3][H:5][D:1][D:5][D:3]   │
└─────────────────────────────────────────────┘
```

**Still TCP underneath** → still suffers from TCP head-of-line blocking. One dropped packet stalls ALL streams until retransmit.

### HTTP/3 (2022)
**QUIC-based - UDP with superpowers**

HTTP/3 is HTTP/2 semantics over QUIC (not TCP). This is the big architectural shift.

**QUIC fundamentals**:

| Feature | TCP | QUIC |
|---------|-----|------|
| Transport | Kernel TCP | Userspace UDP |
| Encryption | Optional (TLS layer) | Mandatory, built-in (TLS 1.3) |
| Handshake | TCP 3-way + TLS = 2-3 RTT | 1 RTT (0-RTT for repeat) |
| Head-of-line | Connection-wide | Per-stream only |
| Connection ID | 4-tuple (IPs + ports) | 64-bit ID (survives IP change) |
| Congestion | In kernel | In userspace (evolvable) |

**Why UDP?**
- TCP is ossified - middleboxes (NATs, firewalls) inspect/modify TCP
- QUIC encrypts almost everything, middleboxes can't interfere
- Userspace means faster iteration (no kernel updates needed)

**Stream independence**:
```
QUIC Connection
┌─────────────────────────────────────────────┐
│  Stream 0: Control                          │
│  Stream 4: GET /index.html ──┐              │
│  Stream 8: GET /style.css ───┼─ INDEPENDENT │
│  Stream 12: GET /app.js ─────┘              │
│                                             │
│  Packet loss on Stream 8?                   │
│  → Only Stream 8 stalls                     │
│  → Streams 4 and 12 continue                │
└─────────────────────────────────────────────┘
```

**Connection migration**: Your phone switches WiFi → cellular? Connection ID stays same, no re-handshake.

**0-RTT resumption**: Return visits can send data immediately with cached keys (with replay attack caveats).

## Process Isolation Model

This is the right call for security and stability. Valet should **never** run user code in its own process.

**Options for the process runner:**

| Approach | Pros | Cons |
|----------|------|------|
| **Unix sockets** | Fast, secure, simple | Unix-only (though we can abstract) |
| **TCP loopback** | Cross-platform | Slightly more overhead |
| **Shared memory + signals** | Fastest possible | Complex, platform-specific |

ASGI is a good reference model — it's already designed for this:

```python
# ASGI interface (conceptually)
async def app(scope, receive, send):
    ...
```

But we'd want a Ritz-native equivalent:

```ritz
// Ritz app interface
pub fn handle(req: http.Request) -> http.Response {
    // ...
}

// Or async/streaming version
pub async fn handle(req: http.Request, respond: fn(http.Response)) {
    // ...
}
```

## The Interface Protocol

Between Valet and the app process, you need a wire format. Options:

1. **Raw HTTP** - Valet just proxies HTTP to the app (simple, but overhead)
2. **ASGI-style** - Scope dict + receive/send channels (Python heritage)
3. **Custom binary** - Minimal overhead, Ritz-native (e.g., msgpack or custom)

For Ritz, I'd lean toward a **custom binary protocol** that maps cleanly to `http.Request`/`http.Response` types — no JSON/dict overhead, just serialize the structs directly.

## Ring Buffer Design

Classic SPSC (Single Producer, Single Consumer) ring buffer:

```
         write_idx
             ▼
┌───┬───┬───┬───┬───┬───┬───┬───┐
│ D │ E │   │   │   │ A │ B │ C │  ← slots
└───┴───┴───┴───┴───┴───┴───┴───┘
                     ▲
                 read_idx

- Producer writes at write_idx, advances it
- Consumer reads at read_idx, advances it  
- When write catches read = FULL
- When read catches write = EMPTY
- Atomic operations on indices, no locks needed
```

**Zero-copy flow:**

1. Valet receives HTTP request, parses headers
2. Writes request metadata to ring slot (just pointers/offsets into shared buffer)
3. Body data? Points directly into shared memory region
4. Bumps write index (atomic)
5. Optionally signals worker (futex/eventfd)
6. Worker wakes, reads slot, processes
7. Writes response to response ring
8. Valet reads response, sends to client

**No copies.** The HTTP body bytes never move — just pointers flying back and forth.

## Memory Layout

```
Shared Memory Region (e.g., 64MB)
┌─────────────────────────────────────────────────────────┐
│  Control Block (4KB)                                    │
│  ┌─────────────────────────────────────────────────────┐│
│  │ req_write_idx │ req_read_idx │ ...                  ││
│  │ resp_write_idx │ resp_read_idx │ ...                ││
│  └─────────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────────┤
│  Request Ring (N slots × slot_size)                     │
│  ┌─────┬─────┬─────┬─────┬─────┬─────┐                 │
│  │ S0  │ S1  │ S2  │ ... │     │     │                 │
│  └─────┴─────┴─────┴─────┴─────┴─────┘                 │
├─────────────────────────────────────────────────────────┤
│  Response Ring (N slots × slot_size)                    │
├─────────────────────────────────────────────────────────┤
│  Data Arena (bulk of memory)                            │
│  ┌─────────────────────────────────────────────────────┐│
│  │ Request bodies, response bodies, headers, etc.      ││
│  │ Allocated in chunks, freed when request completes   ││
│  └─────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────┘
```

## Slot Structure

```ritz
struct RequestSlot {
    id: u64,                    // Request ID for correlation
    method: u8,                 // GET=1, POST=2, etc.
    path_offset: u32,           // Offset into data arena
    path_len: u16,
    headers_offset: u32,        // Serialized headers
    headers_len: u16,
    body_offset: u32,           // Body data (zero-copy!)
    body_len: u32,
    flags: u8,                  // Streaming? Keep-alive? etc.
}

struct ResponseSlot {
    id: u64,                    // Correlates to request
    status: u16,                // HTTP status code
    headers_offset: u32,
    headers_len: u16,
    body_offset: u32,
    body_len: u32,
    flags: u8,                  // Chunked? Complete? Error?
}
```

## Signaling

Ring buffers need a wake mechanism when transitioning empty → has-data:

| Platform | Mechanism |
|----------|-----------|
| Linux | `futex` on control word |
| macOS | `dispatch_semaphore` or `pthread_cond` on shared mem |
| Windows | Named semaphore or `WaitOnAddress` |
| Portable | `eventfd`-style or busy-poll with backoff |

For max performance: **spin briefly, then futex wait**. Most requests complete fast enough that spinning wins.

## Multi-Worker Scaling

Zeus can spawn N worker processes, each with their own ring pair:

```
Valet
  │
  ├──► Zeus Worker 0 (Ring 0)
  ├──► Zeus Worker 1 (Ring 1)
  ├──► Zeus Worker 2 (Ring 2)
  └──► Zeus Worker 3 (Ring 3)

Load balancing: round-robin, least-connections, or work-stealing
```

Or MPSC (Multiple Producer, Single Consumer) if you want one ring with multiple workers reading — but SPSC per-worker is simpler and avoids contention.
