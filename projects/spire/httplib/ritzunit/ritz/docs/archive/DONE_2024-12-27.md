### Session Summary
Implemented proper separate compilation for ritz0. Each `.ritz` module can now be compiled to its own `.o` file and linked with `ld`.

### Key Changes

1. **CLI flag** (`ritz0.py`)
   - Added `--separate` flag for separate compilation mode
   - Functions from main source: `define` (full body)
   - Functions from imports: `declare` only (external reference)

2. **Source file tracking** (`ritz_ast.py`, `import_resolver.py`)
   - Items now track `source_file` attribute for origin tracking
   - Set by import resolver when registering items

3. **Monomorphization fix** (`monomorph.py`)

   - Bug: was losing source tracking during generic specialization

4. **Emitter enhancement** (`emitter_llvmlite.py`)
   - `_should_emit_body()` checks if function should get full body or just declaration
   - Only main source functions get `define`, imported get `declare`

### Results

```bash
# Normal (monolithic): 4491 lines LLVM IR → 25544 byte binary
# Separate (5 modules): 772 lines main + 4 libs → 25672 byte binary (linked)

$ /tmp/grep_linked world /tmp/test.txt
Hello world
Goodbye world
```

### Benefits
- **Incremental compilation**: Only recompile changed modules
- **Smaller compilation units**: Faster individual compiles
- **Linking flexibility**: Can link Ritz code with external object files
- **No duplicate definitions**: Each function defined exactly once across all object files

### Usage
```bash
# Compile library modules
python3 ritz0/ritz0.py ritzlib/sys.ritz --separate --no-runtime -o sys.ll
llc -O2 -filetype=obj sys.ll -o sys.o

# Compile main module
python3 ritz0/ritz0.py examples/11_grep/src/main.ritz --separate -o grep.ll
llc -O2 -filetype=obj grep.ll -o grep.o

# Link
ld -dynamic-linker /lib64/ld-linux-x86-64.so.2 -lc -o grep sys.o io.o str.o args.o grep.o
```

See `ritz0/SEPARATE_COMPILATION.md` for full documentation.

## 2024-12-26: 9+ Character Identifier Bug FIXED ✅

### Session Summary
- **FIXED: NFA lexer 9+ character identifier bug** (GitHub issue #49)
- **Root cause**: `read_file()` used `__builtin_alloca` (stack allocation), causing use-after-stack-free when returning pointer to caller
- **Fix**: Changed to use `mmap` for file buffer allocation

### Root Cause Analysis
The `read_file()` function was using `__builtin_alloca(BUF_SIZE)` to allocate the source file buffer. This allocates on the **stack of read_file()**, and when the function returns, the stack frame is deallocated. The returned pointer becomes a dangling pointer.

With longer identifiers (9+ chars), the NFA lexer's recursive matching explored more branches, pushing more stack frames that overlapped with the file buffer's former location, corrupting the source text.

### The Fix
```ritz
# Before (BUG):
let buf: *i8 = __builtin_alloca(BUF_SIZE)  # Stack allocation - dangling after return!

# After (FIXED):
let buf: *i8 = syscall6(9, 0, BUF_SIZE, 3, 34, -1, 0) as *i8  # mmap - heap allocation
```

### Doctrine Applied
This bug exemplifies both doctrines:
1. **No Concessions Doctrine**: We fixed the root cause instead of renaming functions to 8 chars
2. **Ghost Doctrine**: A bug that silently corrupts memory *will* manifest in mysterious ways later

## 2024-12-26: ritz1 IR Generation Fixes ✅

### Session Summary
- **Fixed `br i1` type mismatch**: Comparisons zext to i32 but `br` needs i1
- **Fixed unreachable code after return**: Blocks ending with return no longer emit trailing `br`
- **Added `*mut` pointer syntax support**: Parser now handles `*mut T` in var/let/param/field types

### Bug Fixes

#### 1. Branch Instruction Type Mismatch
Comparisons (`icmp`) return i1, which ritz1 zext'd to i32 for arithmetic. But `br i1 %cond` expects an i1.

**Fix**: Added `trunc i32 ... to i1` before branch instructions in if/while statements.

#### 2. Unreachable Code After Return
If a then/else block ended with `return`, the emitter still added `br label %end`, creating unreachable code and breaking SSA register numbering.

**Fix**: Track if last statement in block is STMT_RETURN; skip branch emission if so.

#### 3. `*mut` Pointer Syntax
Parser didn't handle `*mut T` syntax for mutable pointers.

**Fix**: Added `parser_check_mut()` helper and updated type parsing in:
- `parser_parse_var_decl()`
- `parser_parse_let_decl()`
- Function parameter parsing
- Struct field parsing

### Known Limitations
1. **No import resolution**: ritz1 parses `import` but doesn't load module contents
2. **All variables treated as i32**: Pointer types parsed but emitter uses i32 for loads/stores
3. **No pointer arithmetic in IR**: Array indexing works but with wrong types

## Session 8: Syscall Builtins (2025-01-XX)

### Goal
Replace extern fn dependencies with pure syscall builtins for bootstrap compatibility.

### Implementation
Added syscall builtin detection and inline assembly emission to `ritz1/src/main_new.ritz`:
- **write(fd: i32, buf: *u8, count: i64) -> i64** - syscall 1
- **read(fd: i32, buf: *mut u8, count: i64) -> i64** - syscall 0
- **open(path: *u8, flags: i32) -> i32** - syscall 2
- **close(fd: i32) -> i32** - syscall 3

### Pattern
Each syscall:
1. Name detection via byte-by-byte string comparison
2. Evaluate all arguments to registers
3. Convert arguments to i64 (sext for integers, ptrtoint for pointers)
4. Emit inline assembly: `call i64 asm sideeffect "syscall", "=&{rax},{rax},{rdi},{rsi},{rdx},~{rcx},~{r11},~{memory}"`
5. Return i64 result (or truncate to i32 for open/close)
