## GC Strategy Decision

**Recommendation: Start with generational, but use a simpler hybrid approach**

The research shows that generational GC requires:
1. **Write barriers** - tracking when old gen objects point to young gen
2. **Remembered sets / card tables** - data structures for inter-generational pointers
3. **Two collection algorithms** - semi-space copying for young gen, mark-sweep for old gen

That's ~2-3x the complexity of pure mark-sweep. BUT there's a smart middle ground that V8 and other engines use:

### Proposed Sage GC Architecture

```
┌─────────────────────────────────────────────────────┐
│                    YOUNG GEN                         │
│    Semi-space copying (Cheney's algorithm)          │
│    Fast bump allocation, frequent collection         │
└────────────────────────┬────────────────────────────┘
                         │ survives 2 GCs
                         ▼
┌─────────────────────────────────────────────────────┐
│                     OLD GEN                          │
│    Mark-sweep (non-moving, simpler)                 │
│    Infrequent collection                            │
└─────────────────────────────────────────────────────┘
```

This is exactly what V8 does - and it's elegant because:
- Young gen uses fast copying collector (most objects die young)
- Old gen uses simpler mark-sweep (they rarely move anyway)
- We still need write barriers, but avoid complex compaction

**Effort estimate**: ~40% more work than pure mark-sweep, but we get 80% of the benefit.

## Updated Project Architecture

Let me draft the full architecture now:

### Register-Based VM Design

```
┌─────────────────────────────────────────────────────┐
│                  REGISTER FILE                       │
│  r0  r1  r2  r3  r4  r5  r6  r7  ... r255           │
│  (256 registers per function frame)                 │
└─────────────────────────────────────────────────────┘

Bytecode format: [opcode: u8][dst: u8][src1: u8][src2: u8]
Example: ADD r0, r1, r2  →  0x01 0x00 0x01 0x02
```

**Why 256 registers?** Fits in a byte, enough for any reasonable function.

### Bytecode Instruction Set (Initial)

| Opcode | Mnemonic | Description |
|--------|----------|-------------|
| 0x00 | `LOAD_CONST` | Load constant into register |
| 0x01 | `ADD` | r[dst] = r[a] + r[b] |
| 0x02 | `SUB` | r[dst] = r[a] - r[b] |
| 0x03 | `MUL` | r[dst] = r[a] * r[b] |
| 0x04 | `DIV` | r[dst] = r[a] / r[b] |
| 0x05 | `MOV` | r[dst] = r[src] |
| 0x06 | `RET` | Return r[src] |
| ... | ... | ... |

## Revised Implementation Phases

### Phase 1: Core Pipeline (MVP)
1. **Lexer** - Tokenize JS source
2. **AST** - Define node types  
3. **Parser** - Build AST from tokens
4. **Bytecode** - Define instruction set
5. **Compiler** - AST → bytecode
6. **VM** - Register-based interpreter
7. **Value** - JS value representation (NaN-boxing?)

**Goal**: `2 + 2` evaluates to `4`

### Phase 2: Variables & Functions
- Local variables (register allocation)
- Global variables
- Function definitions & calls
- Return values

### Phase 3: Control Flow
- `if`/`else`
- `while`/`for` loops
- Comparison operators

### Phase 4: Objects & GC
- Object representation
- Property access
- **Generational GC** (young gen + old gen)
- Write barriers

### Phase 5: Full Runtime
- Arrays, strings, closures
- Built-in functions (`console.log`, `Math.*`)
- Error handling (`try`/`catch`)

### Phase 6: ES6+
- Arrow functions, `let`/`const`
- Classes, destructuring
- Iterators, generators
- Eventually: async/await
