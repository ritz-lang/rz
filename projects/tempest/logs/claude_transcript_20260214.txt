## Key Design Questions

### 1. **What's our target scope?**

Options:
- **A) Minimal Viable Browser** - Load and render HTML/CSS pages
- **B) Full Featured** - Tabs, history, bookmarks, downloads, dev tools
- **C) Full + JS** - Everything above + JavaScript execution

### 2. **Platform targets?**

Options:
- **A) Harland-first** - Build for Harland/Prism display server
- **B) Linux framebuffer** - Direct `/dev/fb0` rendering (simpler)
- **C) X11/Wayland** - Standard desktop Linux (wider audience)
- **D) All of the above** - Abstract rendering backend

### 3. **Build order strategy?**

The dependency chain is: Tempest → Lexis → Iris → Prism

Options:
- **A) Bottom-up** - Build Prism → Iris → Lexis → Tempest
- **B) Top-down stubs** - Build Tempest shell with stub deps, fill in later
- **C) Parallel** - Define interfaces, build all simultaneously
- **D) Network-first** - HTTP client (Valet-client), then HTML parsing

### 4. **Multi-process architecture?**

Modern browsers use process isolation:
- Chrome model: Browser process + renderer processes + GPU process
- Firefox model: Similar but with more shared memory

For Tempest:
- **A) Single process** - Simple, good for embedded/Harland
- **B) Multi-process** - Each tab isolated (security + stability)
- **C) Hybrid** - Main process + sandbox for JS execution

### Key Design Decisions from Iris Session

1. **Tempest owns the DOM** - Iris receives StyleEvents describing changes
2. **Iris builds a separate render tree** - Mirrors visible DOM, optimized for layout
3. **Frame-based batching** - Mutations batched at 60fps, not immediate layout
4. **Iris does hit testing** - Since it owns render tree with positions
5. **Iris handles scrolling** - Fast path at 60fps, notifies Tempest for JS `onscroll`

### Key Interfaces

**Tempest → Iris (StyleEvent):**
```ritz
enum StyleEvent
    NodeCreated { id: NodeId, parent: NodeId, index: u32, tag: Tag }
    NodeRemoved { id: NodeId }
    NodeMoved { id: NodeId, new_parent: NodeId, new_index: u32 }
    StyleChanged { id: NodeId, style: ComputedStyle }
    TextChanged { id: NodeId, text: StrView }
```

**Iris → Tempest (Callbacks):**
```ritz
struct HitTestResult
    node_id: NodeId
    point: Point            # Relative to node

struct ScrollNotification
    node_id: NodeId         # Scrolling container
    offset: Point           # New scroll position
```

### Prism API Contract (Stable)

Iris uses these types from Prism:
- `Point { x: i32, y: i32 }`
- `Size { width: u32, height: u32 }`
- `Rect` with rich geometry API
- `Color { r, g, b, a: u8 }`
- `DrawContext` - fill_rect, stroke_rect, draw_glyphs, draw_image, etc.
- `Layer` - for CSS transforms, opacity, scroll containers
- `GlyphRun` - shaped text from Angelo

## Tempest's Role

Given these decisions, Tempest is the **orchestrator**. It:

1. **Owns the DOM tree** - The canonical source of page structure
2. **Coordinates parsing** - Uses Lexis for HTML/CSS parsing
3. **Runs JavaScript** - Uses Sage for execution, handles DOM mutations
4. **Manages navigation** - URLs, history, back/forward
5. **Manages tabs** - Multiple browser contexts
6. **Dispatches events** - Receives from Iris, routes to DOM event handlers
7. **Sends StyleEvents** - Tells Iris what changed

### Tempest Does NOT:
- Parse HTML/CSS directly (Lexis)
- Execute JavaScript directly (Sage)
- Layout or paint (Iris)
- Draw pixels (Prism)

The HTTP library is a **mechanism-not-policy** wire protocol library that Tempest will use as its HTTP client. Here's what it provides:

| **HTTP/2** | ✓ | Binary framing, HPACK compression, stream multiplexing, flow control |
| **HTTP/3** | ✓ | QUIC transport, QPACK compression, stream management |

### Key Files for Tempest

| File | Purpose |
|------|---------|

| `lib/h1_request.ritz` | Parse HTTP/1.x requests (for servers, but useful for testing) |
| `lib/h1_response.ritz` | Parse HTTP/1.x responses (what Tempest needs as a client!) |
| `lib/h1_writer.ritz` | Serialize HTTP/1.x requests (what Tempest sends) |
| `lib/async_http.ritz` | io_uring-based async HTTP operations |
| `lib/h2_*.ritz` | HTTP/2 frame/stream handling |
| `lib/h3_*.ritz` | HTTP/3 frame/stream handling |
| `lib/quic_*.ritz` | QUIC transport layer |

### Integration Points for Tempest

1. **HTTP Client (outgoing requests)**:
   - Use `h1_write_request_line()` + `h1_write_header()` to construct requests
   - Use `h1_parse_response()` to parse server responses
   - `AsyncHttpConn` manages connection state with io_uring

2. **Protocol Negotiation**:
   - Start with HTTP/1.1 over TLS (via cryptosec)
   - ALPN negotiation for HTTP/2 (`h2`)
   - UDP + QUIC for HTTP/3 (full implementation available)

3. **Dependencies**:
   - **cryptosec** - TLS 1.3 (integration TODO in http library)
