[2026-02-15T01:57:30.026279] Claude: Issue #163 is for **multi-statement blocks in match arms**.

Right now, ritz0 only allows a single expression after `=>` in match arms:

```ritz
# ✅ This works - single expression

    Ok(x) => x + 1
    Err(_) => 0

# ❌ This fails - multiple statements
match @self.loca
    Some(loca) =>
        let offset = loca.get_offset(glyph_id)?
        let length = loca.get_length(glyph_id)?
        if length == 0
            return Some(GlyphOutline.empty())
        # ... more logic
    None => None
```

The parser currently calls `parse_expr()` for match arm bodies, which only accepts a single expression. It needs to recognize an indent after `=>` and call `parse_block()` instead to allow multiple statements.

**Why it matters**: Angelo's font.ritz uses this pattern extensively (~11 occurrences across 4 files) for complex glyph loading logic. Without this, we can't compile Angelo, which means no proper TTF font rendering.

The fix should be straightforward in the parser - when `parse_match_arm()` sees a newline+indent after `=>`, parse a block instead of a single expression.
