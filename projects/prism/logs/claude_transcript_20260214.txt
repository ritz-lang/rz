## Double vs Triple Buffering Deep Dive

### Double Buffering
```
┌─────────────┐     ┌─────────────┐
│  Front      │     │  Back       │
│  Buffer     │◄────│  Buffer     │
│  (display)  │ swap│  (drawing)  │
└─────────────┘     └─────────────┘
```

**How it works:**
- App draws to back buffer
- When frame is ready, swap buffers atomically
- Display shows front buffer while app draws to (new) back buffer

**Problem:** If the app is slow, the display waits. If vsync is on, the app blocks until the swap happens. This can cause **stuttering** — if you miss a frame, you wait a whole refresh cycle.

### Triple Buffering
```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Front      │     │  Back       │     │  Pending    │
│  Buffer     │◄────│  Buffer     │◄────│  Buffer     │
│  (display)  │     │  (ready)    │     │  (drawing)  │
└─────────────┘     └─────────────┘     └─────────────┘
```

**How it works:**
- App draws to pending buffer (never blocks)
- When done, pending becomes back (ready to display)
- At vsync, back swaps to front
- App immediately starts on a new pending buffer

**Benefit:** The app never waits. If it finishes a frame before vsync, it starts the next frame immediately. **Smoother animations** because you're always showing the most recent complete frame.

**Cost:** One extra buffer per window (more memory), slightly more latency (but only ~1 frame).

### Damage Tracking (Orthogonal to Buffer Count)

This is different! Damage tracking is about **what regions changed**:

```
┌─────────────────────────────┐
│  Window A (unchanged)       │ ← Skip redraw
├─────────────────────────────┤
│  Window B    ┌────────┐     │
│              │ DIRTY  │     │ ← Only redraw this rect
│              └────────┘     │
├─────────────────────────────┤
│  Window C (unchanged)       │ ← Skip redraw
└─────────────────────────────┘
```

**Key insight:** Whether you use double or triple buffering, damage tracking reduces *what* you composite each frame. They compose well together.

### For Prism + Animations

I'd recommend:

1. **Triple buffering for the compositor** — Animations need to be buttery smooth. Triple buffering means we never stall waiting for vsync.

2. **Per-window damage tracking** — Windows report "dirty rectangles" when they update. If Window A is static while Window B animates, we only re-composite Window B's region.

3. **Occlusion culling** — If Window A is fully behind Window B, don't even ask Window A to redraw. This is the "obscured windows" optimization you mentioned.

```
┌─────────────────────────────────────────────────────┐
│                   COMPOSITOR                         │
│                                                      │
│  ┌─────────────┐  Window submits:                   │
│  │ Window      │  - buffer handle (shared memory)   │
│  │ Buffers     │  - damage rects                    │
│  │ (per-app)   │  - "frame complete" signal         │
│  └─────────────┘                                    │
│         │                                           │
│         ▼                                           │
│  ┌─────────────────────────────────┐               │
│  │     Damage Aggregation          │               │
│  │  - Union all dirty rects        │               │
│  │  - Subtract occluded regions    │               │
│  │  - Compute minimal redraw       │               │
│  └─────────────────────────────────┘               │
│         │                                           │
│         ▼                                           │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐   │
│  │   Front     │ │   Back      │ │  Pending    │   │
│  │   Buffer    │ │   Buffer    │ │  Buffer     │   │
│  └─────────────┘ └─────────────┘ └─────────────┘   │
│       ▲              Triple buffering               │
```

## X11 vs Wayland Rendering Models

### X11: Server-Side Rendering (mostly)

```
┌─────────────┐                    ┌─────────────┐
│   Client    │  "Draw rect at    │   X Server  │
│   (App)     │  x,y,w,h" ───────▶│             │
│             │                    │  Actually   │
│  Sends      │  "Render text     │  draws to   │
│  commands   │  'Hello'" ───────▶│  framebuffer│
└─────────────┘                    └─────────────┘
```

**How it works:**
- Client sends **drawing commands** (draw line, fill rect, render text)
- X server executes commands and renders to the window
- Network-transparent by design (commands are small, serializable)

**Pros:** Great for remoting, low client memory usage
**Cons:** Chatty protocol, server is complex, hard to do modern effects

### Wayland: Client-Side Rendering

```
┌─────────────┐                    ┌─────────────┐
│   Client    │  1. Allocate      │  Wayland    │
│   (App)     │     shared mem    │  Compositor │
│             │                    │             │
│  Renders    │  2. Render to     │  Composites │
│  to buffer  │     buffer        │  buffers    │
│             │                    │  together   │
│             │  3. "Buffer       │             │
│             │     ready" ──────▶│             │
└─────────────┘                    └─────────────┘
```

**How it works:**
- Client allocates shared memory buffer
- Client renders using whatever (OpenGL, Cairo, software)
- Client signals "I'm done, here's my buffer + damage rects"
- Compositor just blits buffers together

**Pros:** Simple compositor, clients have full control, modern effects easy
**Cons:** Harder to remote (need to ship pixels), more client memory

## The Async Scanline Model

I love this idea. Let me formalize it:

```
┌─────────────────────────────────────────────────────────────────────┐
│                    ASYNC SCANLINE COMPOSITOR                         │
│                                                                      │
│  Frame N displaying    Frame N+1 ready    Frame N+2 drawing         │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐      │
│  │█████████████████│  │█████████████████│  │▓▓▓▓▓▓▓▓▓░░░░░░░│      │
│  │█████████████████│  │█████████████████│  │▓▓▓▓▓▓▓▓▓░░░░░░░│      │
│  │█████████████████│  │█████████████████│  │▓▓▓▓▓▓▓▓▓░░░░░░░│      │
│  │█████████████████│  │█████████████████│  │───scanline────│ ←     │
│  │█████████████████│  │█████████████████│  │░░░░░░░░░░░░░░░│      │
│  │█████████████████│  │█████████████████│  │░░░░░░░░░░░░░░░│      │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘      │
│        FRONT               BACK                PENDING              │
│                                                                      │
│  ▓ = already rendered this frame                                    │
│  ░ = not yet rendered (safe to update)                              │
│  ─ = virtual scanline position                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### Rules:

1. **Virtual scanline** advances at frame_rate * screen_height pixels per second
2. **Components above scanline** — too late, wait for next frame
3. **Components below scanline** — safe to render now
4. **Dirty regions** — components invalidate themselves, get queued
5. **Render queue** — sorted by Y position, workers pull from queue

```ritz
struct RenderQueue
    dirty_regions: Vec<DirtyRegion>
    scanline_y: u32                    # current virtual scanline
    frame_start: u64                   # timestamp

struct DirtyRegion
    bounds: Rect
    component: ComponentId
    priority: u8                       # higher = render first

fn can_render_now(region: DirtyRegion, queue: RenderQueue) -> bool
    # Region's top edge must be below the scanline
    region.bounds.y > queue.scanline_y

fn render_loop(queue:& RenderQueue)
    loop
        # Update virtual scanline based on time
        let elapsed = now() - queue.frame_start
        let progress = elapsed / FRAME_DURATION
        queue.scanline_y = (progress * SCREEN_HEIGHT) as u32
        
        # Find regions we can still render this frame
        for region in queue.dirty_regions
            if can_render_now(region, queue)
                # Spawn async render task
                spawn_render(region)
                
        # If scanline reached bottom, swap buffers
        if queue.scanline_y >= SCREEN_HEIGHT
            swap_buffers()
            queue.frame_start = now()
            queue.scanline_y = 0
```

### Parallel Rendering

Multiple workers can render simultaneously if regions don't overlap:

```ritz
fn can_render_parallel(a: Rect, b: Rect) -> bool
    not a.intersects(b)

# Worker pool pulls non-conflicting regions
fn worker_thread(queue:& RenderQueue)
    loop
        let region = queue.claim_non_conflicting()
        match region
            Some(r) => 
                render_region(r)
                queue.mark_complete(r)
            None => 
                yield()  # nothing available, wait
```

## Remote Display Design

If we want SSH tunneling to work well, the **command mode is key**:

```
┌─────────────┐          SSH Tunnel           ┌─────────────┐
│   Remote    │                               │   Local     │
│   Client    │  RenderCommands ────────────▶ │   Prism     │
│             │                               │             │
│  (runs on   │  WindowEvents  ◀──────────── │  (displays  │
│   server)   │                               │   locally)  │
└─────────────┘                               └─────────────┘
```

Commands are small and serializable. Compare:

| Approach | Per-frame bandwidth (1080p window) |
|----------|-----------------------------------|
| Buffer mode (pixels) | ~8MB uncompressed, ~100KB compressed |
| Command mode | ~1-10KB |

For remote display, command mode wins by 10-100x.

**Design implication:** The command protocol should be designed to be serializable from day one. No pointers in commands — use handles/IDs.

### Who does the drawing?

**Both, depending on mode:**

| Mode | Who renders | When to use |
|------|-------------|-------------|
| Buffer | Client | Complex custom rendering (games, browser canvas) |

### Event model?

**Event-driven, not polling:**

```ritz
# Widget main loop (pseudo-code)
fn widget_main(cap: WindowCapability)
    loop
        # Block until event arrives (async/io_uring)
        let event = cap.event_ring.wait_read()
        
        match event
            WindowEvent.KeyPress(key, mods) =>
                handle_key(key, mods)
                cap.command_ring.write(Invalidate { region: dirty_rect })
                
            WindowEvent.FrameCallback(frame) =>
                # Good time to update animations
                update_animations(frame)
                
            WindowEvent.Close =>
                break
```

No busy-loop. Widget sleeps until Prism (or Harland) wakes it.

### MockBackend Design

```ritz
# The mock backend gives us a pixel buffer we can inspect
struct MockBackend
    width: u32
    height: u32
    buffers: [3]Vec<u32>          # triple buffer, RGBA pixels
    current: u32                   # which buffer is "displayed"
    
    # For testing
    frame_count: u64
    damage_history: Vec<Rect>     # track what regions were redrawn

impl Backend for MockBackend
    fn get_buffer(self:&) -> @&[u32]
        @&self.buffers[self.pending_index()]
    
    fn swap(self:&)
        self.current = (self.current + 1) % 3
        self.frame_count += 1
    
    fn dimensions(self) -> (u32, u32)
        (self.width, self.height)

# Test helpers
impl MockBackend
    fn pixel_at(self, x: u32, y: u32) -> Color
        let idx = y * self.width + x
        Color.from_u32(self.buffers[self.current][idx])
    
    fn assert_rect_color(self, rect: Rect, expected: Color)
        for y in rect.y..rect.y + rect.height
            for x in rect.x..rect.x + rect.width
                assert self.pixel_at(x, y) == expected
    
    fn assert_region_untouched(self, rect: Rect, since_frame: u64)
        # Check damage history - rect should not appear
        for damage in self.damage_history
            if damage.frame >= since_frame and damage.rect.intersects(rect)
                fail("Region was touched when it shouldn't have been")
```
