**Two approaches for data:**

| Approach | Pros | Cons |
|----------|------|------|
| **Partitioned** (each worker owns keys) | Simple, no sync needed | Client must know which worker has key |
| **Replicated** (eventual consistency) | Any worker can answer | Need sync mechanism, more memory |

**Eventual consistency options:**
- **Gossip protocol** - Workers broadcast mutations to peers
- **Write-ahead log** - Shared WAL that all workers replay
- **CRDT-style** - Conflict-free data types (counters, sets)

**My recommendation for Tome:**

Start with **partitioned + SO_REUSEPORT**:
1. Each worker has its own Store
2. Kernel load-balances connections across workers
3. Sticky connections (client stays with one worker)
4. Simple, high-performance, no coordination overhead

Then add **optional replication** for read scaling:
- `REPLICATE key` command to sync specific keys
- Async gossip for "hot" keys
- Eventually consistent reads, strongly consistent writes (per-worker)

This matches Redis Cluster's model but simpler - and as you said, same semantics locally or distributed.

## Addressing Your Question: "Why do they take *u8 instead of StrView?"

Great observation. The current Store API uses `*u8` for keys:

```ritz
pub fn store_get(s: *Store, key: *u8) -> *u8

```

This is a **legacy design** from before RERITZ. In RERITZ, we should prefer `StrView` which is:
- Zero-copy (pointer + length)
- Length-prefixed (no null-terminator scan needed)
- More type-safe (can't accidentally pass random memory)

**Better API would be:**
```ritz
pub fn store_get(s: *Store, key: StrView) -> StrView
pub fn store_set(s: *Store, key: StrView, value: StrView)
```

Then callers can use:
```ritz
store_set(store, "mykey", "myvalue")  # Direct StrView literals
```
