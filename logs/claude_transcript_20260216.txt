## Now: Filesystem Architecture Discussion

You raised an incredibly important point: **we've been implementing without a proper design**. Let me present my thoughts on the "everything is a resource in a namespace" approach:

### The Traditional Model: Everything is a File (Unix/Linux)

```
/dev/sda          # Block device
/dev/tty0         # Terminal
/proc/1/status    # Process info (synthetic)
/sys/class/net/   # Network devices (synthetic)
/home/user/file   # Actual file
```

Unix shoehorns everything into the file abstraction. This works but creates awkwardness:
- `read()` on a directory fails, you need `readdir()`
- `ioctl()` for device-specific operations (magic numbers)
- `/proc` and `/sys` are fake filesystems that don't persist
- Sockets need `socket()` not `open()`

### The Ritz Model: Everything is a Resource in a Namespace

What if the namespace is the **primary abstraction**, and files are just one type of resource?

```ritz
# A namespace is a capability-controlled hierarchy
struct Namespace
    root: ResourceRef          # Root resource
    parent: Option<NamespaceRef>  # Optional parent namespace
    
# Resources have types, not just "file" vs "directory"
enum ResourceKind
    File(BlobRef)              # Content-addressable file
    Directory(Vec<DirEntry>)   # Collection of named resources
    Channel(ChannelEndpoint)   # IPC channel
    Process(Pid)               # Process handle
    Device(DeviceCap)          # Device driver endpoint
    Socket(SocketRef)          # Network socket
    Mount(NamespaceRef)        # Mounted namespace
```

### Key Insight: Namespaces as Capability Boundaries

```
/                           # System root namespace (kernel only)
├── boot/                   # Boot resources (initramfs)
├── dev/                    # Device endpoints (capability-gated)
│   ├── disk0               # VirtIO disk (requires CAP_DISK_0)
│   └── serial              # Serial port (requires CAP_SERIAL)
├── proc/                   # Process namespace (synthetic)
│   └── {pid}/              # Per-process view
│       ├── caps            # Capability list for this process
│       └── ns              # This process's namespace root
├── sys/                    # System configuration
└── home/                   # User namespaces
    └── {user}/             # User-specific namespace root
```

### The Boot Sequence I'd Propose

1. **Kernel initializes** with minimal embedded filesystem:
   ```
   /boot/init              # PID 1 binary (capability broker)
   /boot/config            # Initial capabilities configuration
   ```

2. **Init process starts** with full capabilities:
   ```ritz
   # Init has capability to the entire namespace
   let root_cap: Cap<Namespace> = kernel_provided_root_cap()
   ```

3. **Init reads config** and spawns services:
   ```ritz
   # Start device manager with device caps only
   let devmgr_caps = derive(root_cap, "/dev")
   spawn("/boot/devmgr", caps: [devmgr_caps])
   
   # Start filesystem server with storage caps
   let fs_caps = derive(root_cap, "/mnt") + derive(root_cap, "/dev/disk0")
   spawn("/boot/fsd", caps: [fs_caps])
   ```

4. **Services bind to namespace paths**:
   ```ritz
   # The filesystem server mounts itself at /mnt
   namespace_bind("/mnt", my_namespace)
   ```

### Minimal Boilerplate: The "Portable Getpid" Model Extended

Your `portable_getpid.elf` approach is beautiful. What if ALL programs work this way?

```ritz
# A userspace program's "view" is just the capabilities it has
# No global filesystem, no ambient authority

fn main(caps: CapSet) -> i32
    # If we have a file cap, we can read the file
    match caps.get::<File>("config") {
        Some(f) => process_config(f)

    }
    
    # If we have a channel cap, we can talk to the service

        Some(ch) => ch.send("Started")

    }

```

   - Is `/dev/disk0` always there, or only visible if you have the cap?

3. **User namespaces**
   - Can a user's namespace be completely separate from system?

## Separating Storage Representation from Resource Hierarchy

**You're 100% right** - we've conflated two things:

1. **Goliath** (on-disk storage) - How files/blobs are persisted to block devices
2. **Resource Namespace** (in-memory) - How we view/navigate the resource tree

### Current Architecture
```
kblob.ritz     → Content-addressed blob storage (hash → data)
knamespace.ritz → Path → inode mapping (in-memory tree)
vfs.ritz       → File descriptors, syscall layer
```

### What Should Happen

**Ephemeral objects** (pipes, sockets, devices) should:
- Exist in the namespace tree (visible at `/dev/pipe0`, `/proc/123/fd/3`)
- Have **NO** Goliath backing (pure in-memory)
- Be cleaned up on process exit or explicitly

**Persistent objects** should:
- Exist in namespace tree AND have Goliath backing
- Survive reboots

```
Resource Namespace (knamespace)
  └── /
      ├── etc/         ← backed by Goliath
      │   └── hostname ← backed by Goliath
      ├── dev/         ← namespace-only (ephemeral)
      │   ├── disk0    ← represents VirtIO block device
      │   └── null     ← represents kernel null device
      ├── proc/        ← namespace-only (dynamic)
      │   └── self/    ← changes per-process
      └── tmp/         ← could be either (tmpfs vs persist)
```

## Capability-Based Visibility

**Brilliant insight on visibility vs access:**

```
Directory entry types:
  - VISIBLE_READABLE  - Can see it, can read it
  - VISIBLE_BLOCKED   - Can see it exists, cannot access contents  
  - HIDDEN            - Cannot see it at all
```

For `/dev/`:
```
/dev/
  ├── null     [VISIBLE_READABLE]  - everyone can use
  ├── disk0    [VISIBLE_BLOCKED]   - only block device drivers
  └── kmem     [HIDDEN from non-kernel] - doesn't even appear in ls
```

This makes debugging way easier - you can SEE `/dev/disk0` exists even if you can't touch it.

## Sudo / Privilege Escalation

**This is where Harland can really shine over Linux.**

Instead of `setuid` (which is terrifyingly dangerous), use **capability delegation**:

```
# Instead of setuid:
Process A has: {CAP_READ_FILE, CAP_WRITE_FILE}
Process A wants: {CAP_ADMIN_USERS}

# Harland approach:
1. Process A calls privileged service via IPC
2. Service (the "capability gatekeeper") verifies:
   - Is caller authenticated?
   - Does caller have "may_request_admin" capability?
   - Is request within policy bounds?
3. Service performs privileged operation on A's behalf
   OR grants A a time-limited capability token
```

**No setuid at all.** Instead:
- Privileged daemons run with full caps at boot
- Regular processes request actions via IPC
- Audit trail of every privilege escalation

## Mount Semantics & Namespace Composition

```
# Simple approach: namespace overlays
mount("/shared-volume", "/mnt/shared", {
    source: "virtio://device1/volume0",
    rights: CAP_READ | CAP_WRITE,
})

# What happens:
1. Create new namespace subtree rooted at /mnt/shared
2. Backend: VirtIO paravirt device → block layer → Goliath
3. Each process sees the mount according to its namespace view
```

**Cross-namespace copying** could be:
- Direct memory mapping (zero-copy between Harland processes)
- Block-level copy for hypervisor shared storage
- Network protocol for truly remote namespaces

## Hypervisor Isolation (Future Docker/K8s)

The path forward:
1. **Harland-on-Harland**: Nested capabilities, nested namespaces
2. **Linux guests**: Para-virtualized with VirtIO
3. **Shared volumes**: Special namespace nodes backed by hypervisor-level shared memory

The key insight: **Capabilities compose hierarchically.** A nested Harland can only see what its parent grants it access to.
